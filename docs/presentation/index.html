The companion repo is the reference implementation — this deck explains the choices behind it.

Assumptions I'm working from

  • Scale. 50–200 users initially (one agency). Steady-state, not spiky.
  • Tenancy. Single tenant on day 1; design admits multi-tenant later.
  • Region. UK/EU data residency presumed (marketing data is personal data).
  • Latency. Sub-300ms p95 for screen loads is enough; this is not a trading app.
  • Realtime. Not a launch requirement; we'll model commands/events so it bolts on cleanly.
  • Hosting. Azure preferred (the .NET stack & likely client estate) — agnostic where it costs us little.

Slide 3 of the deck — these are inputs, not conclusions. Each becomes a question on day one (see slide 19).

Technology choices & the trade-offs

LayerChoiceWhy & trade-off
Frontend Angular 18 (standalone, signals) Batteries included, strict TS by default. Heavier bundle than React — fine at this scale.
Backend .NET 8 minimal APIs LTS until Nov 2026, fast, mature ecosystem. We can swap to Controllers if the team prefers.
Database PostgreSQL Open, mature, JSONB if we need flexibility. SQL Server would also work — Postgres is simply cheaper and more portable.
Auth OIDC + JWT bearer Works with whatever IdP the agency already has. We are not in the auth business.
Runtime Azure Container Apps Managed scale-to-zero, no Kubernetes ops overhead. Trade: vendor coupling — abstracted behind IaC.
Observability OpenTelemetry Standardised, swappable backend (App Insights / Honeycomb / Grafana Cloud).
CI/CD GitHub Actions + Bicep Same repo, same review surface. Bicep over Terraform when we're Azure-only.

How the architecture addresses non-functionals

Security

  • OIDC → JWT bearer, short-lived tokens, refresh on the gateway.
  • Secrets in Key Vault via managed identity — none in env files.
  • WAF in front, dependabot + Snyk on PRs, OWASP-aware code review checklist.
  • Tenant scoping enforced at the repository layer.

Scalability

  • Stateless API → horizontal scale on CPU/RPS metrics.
  • Database read replica for reporting workloads.
  • Redis for hot reads (board view, user lookups).
  • Cursor pagination on list endpoints from day one.

Reliability

  • Health probes (liveness vs readiness).
  • Blue/green deploys with automated rollback on SLO burn.
  • Database PITR + nightly logical backups off-region.
  • Retries with jitter on outbound calls; circuit breaker on third parties.

Observability

  • Structured logs (Serilog) with traceId on every request.
  • OpenTelemetry traces from browser → API → DB.
  • RED metrics + SLOs (latency, error rate, throughput) per endpoint.
  • Alert on burn rate, not threshold spikes.

2 · Delivery plan

Phased roadmap: prototype → launch

Now

Phase 0 — Foundation

  • Reference repo, ADRs, environments wired
  • Discovery workshops (users, integrations, compliance)
  • Auth provider & hosting confirmed
Wk 1–3

Phase 1 — MVP

  • Auth, multi-user, tasks, assignment, due dates
  • CI/CD, staging, structured logs, traces
  • Manual UAT with 3 client users
Wk 4–6

Phase 2 — Beta

  • Comments, notifications, attachments
  • Real-time updates (SignalR)
  • Performance pass, accessibility audit
Wk 7–9

Phase 3 — Launch

  • Security review, pen test remediation
  • SLOs + alerting in place, runbooks written
  • Phased rollout (10% → 50% → 100%)
Wk 10+

Phase 4 — Iterate

  • Feedback-driven backlog
  • Tech-debt budget (10–15% per sprint)
  • Quarterly architecture review

Team composition

Core, full-time

  • Tech lead — me. Code, architecture, stakeholder management.
  • 2× full-stack engineers (mid/senior). Vertical-slice delivery.
  • Senior frontend / UX-leaning engineer. Pattern library, accessibility, performance.

Shared, part-time

  • DevOps / platform (~50%) — IaC, pipelines, on-call setup.
  • Designer (~25%) — flows, design tokens, accessibility review.
  • QA (~50% from Phase 2) — exploratory + test automation tooling.

Client-side

  • Product owner with real authority on scope/priority.
  • 2–3 pilot users for weekly demos and UAT.

Why this shape

  • Small enough to fit one stand-up; big enough for sustained pace.
  • Seniority spread enables mentorship without bottlenecks.
  • Specialists arrive when their work is the bottleneck — not before.

Top risks & how we'll mitigate them

RiskLikelihood / impactMitigation
Scope creep — "just one more" High / Medium Strict MVP definition, change-control via PO, weekly demos surface trade-offs early.
Auth provider undecided by client Medium / High OIDC abstraction means we can swap; day-1 question. Default to Entra ID if undecided in week 1.
Data migration from current ad-hoc workflows Medium / High Discovery workshop in week 1; if non-trivial, build an importer as Phase 2, not as a launch blocker.
Hidden compliance requirement (GDPR, ICO) Low / High Privacy review with legal in week 1; design data model with PII flags and export/delete from day one.
Realtime becomes a launch must-have Medium / Medium API designed around commands & queries; SignalR layers on without changing the data model.
Key person dependency Medium / High Pair programming, rotating PR reviewers, ADRs in repo, runbooks in repo.

How I'll surface delivery issues early

Cadence

  • 2-week sprints, demo to PO + pilot users on Friday.
  • Daily 15-minute stand-up — risks/blockers first, status second.
  • Weekly written status to client: RAG, top 3 risks, asks.

Engineering metrics

  • DORA: lead time, deploy frequency, MTTR, change-failure rate.
  • PR cycle time & review latency — visible on a dashboard.
  • Flaky-test budget enforced (no green-on-retry merges).

Product signals

  • Burn-up of MVP scope vs target launch.
  • Weekly tag-up with pilot users for qualitative feedback.
  • Post-deploy error-rate & latency dashboards reviewed each Monday.

Surfacing problems

  • RAG status is honest, not optimistic — "amber" early beats "red" late.
  • If a sprint slips, the deliverable changes, not the date — and we say so.
  • Blameless post-mortems for incidents > 30 min — written, shared, no individual named.

3 · Technical leadership

Setting the team up for success

Ways of working

  • Trunk-based dev, short-lived branches, daily merges.
  • Pair on anything risky or new; mob occasionally for shared learning.
  • Tech-debt time blocked weekly — not optional.
  • Decisions documented as ADRs in repo, not in chat history.

Technical standards

  • Strict TypeScript, nullable reference types, warnings-as-errors.
  • OpenAPI is the API contract; client types generated from it.
  • Every new endpoint ships with at least one integration test.
  • Logs are structured; no Console.WriteLine.

Code review

  • 1 approver for low-risk; 2 for prod paths, schema changes, or auth.
  • Author writes the PR description; reviewer reads it before the diff.
  • Reviews are timely (target: 4 working hours) — slow review kills flow.
  • Comments separate blocking, nit, and question.

Definition of Done

  • Tests written and passing locally and in CI.
  • Deployed to staging; observable in logs/traces.
  • Accessible (WCAG 2.1 AA spot-check) and documented if user-facing.
  • Feature flag in place if rollback isn't trivial.

"The PO wants this feature now — but it's risky"

  1. Acknowledge the goal. The feature exists because of a real outcome the PO needs. Start there, not at "no."
  2. Make the risk concrete. What breaks? Who's affected? What's the blast radius if it goes wrong? Numbers, not adjectives.
  3. Offer options, not roadblocks. A smaller-scoped version. A time-boxed spike. A flag-protected rollout. Always two paths.
  4. Quantify the cost of "fast" vs "safe." Days, not feelings. The PO can choose with eyes open.
  5. If they still push: write the decision and risks as an ADR, get explicit sign-off, ship it behind a kill-switch with heightened monitoring.
  6. Never: say "no" as the engineer alone, or "yes" without the trade-off visible to whoever owns the budget.

What I'd ask on day one

About the users

  • Who actually uses this — roles, team sizes, technical comfort?
  • What does a busy day look like for them today?
  • Which of the chaos (email/calls/meetings) do they actually want to replace, vs supplement?

About the data

  • Where does work live today? Spreadsheets, inboxes, project tools?
  • Does anything need importing on day one — or do we start fresh?
  • Are clients (the agency's clients) ever exposed to this app?

About hosting & identity

  • Existing cloud provider & identity (Entra, Google Workspace, Okta)?
  • UK/EU data residency required? PII categories present?
  • Any existing security or platform standards I should inherit?

About launch & success

  • What does v1 success look like — measurable?
  • Who decides scope when there's pressure to add more?
  • What's the budget for ongoing operation post-launch?

How I'd answer them in the first two weeks

  • Week 1 — workshops. 90-minute sessions with each user role; observe a typical day on-site or shadow-call if remote.
  • Week 1 — written discovery. Existing docs, policies, integration touch-points; security/IT alignment meeting.
  • Week 2 — wireframes. Low-fi mockups validated with pilot users before any further code beyond the prototype.
  • Week 2 — ADRs published. Auth provider, hosting region, data model approach — all written down and signed off.
  • Continuous — assumption log. Anything we couldn't answer is logged with a target date for resolution.

Concerns I'd raise about the brief & prototype

  • "Production-ready" is undefined. Need explicit acceptance criteria — uptime, security, accessibility, observability targets.
  • No identity model in the prototype. Single-user code paths look simple; real auth + tenant scoping reshape almost every endpoint. We'd front-load this.
  • No clear constraint on hosting or compliance. Marketing data is personal data — GDPR is in scope by default.
  • The chaos to replace is bigger than tasks. Email/calls/meetings imply notifications, mentions, summaries — clarify what's in v1 vs Phase 2.
  • "Coordinate" is a multi-app workflow. Slack/Teams integration may be unavoidable — worth confirming in week 1.

In summary

Production-ready isn't a checkbox.

It's a small team shipping every week, behind feature flags, with the trade-offs written down where everyone can see them.
