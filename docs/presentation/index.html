Assumptions I'm working from

Scale

Around 50 to 200 users initially, for one agency. Steady-state, not spiky.

Tenancy

Single tenant on day one. The design admits multi tenant later.

Region

UK and EU data residency presumed. Marketing data is personal data.

Latency

Sub 300ms p95 for screen loads is plenty. This is not a trading app.

Realtime

Not a launch requirement. We model commands and events so it bolts on cleanly.

Hosting

Azure preferred (matches the .NET stack). Agnostic where it costs us little.

These are inputs, not conclusions. Each becomes a day one question (see slide 16).

Technology choices & the trade-offs

LayerChoiceWhy and trade-off
Frontend Angular 18 (standalone, signals) Batteries included, strict TS by default. Heavier bundle than React, fine at this scale.
Backend .NET 8 minimal APIs LTS until Nov 2026, fast, mature ecosystem. Swap to Controllers if the team prefers.
Database PostgreSQL Open, mature, JSONB if we need flexibility. SQL Server would also work; Postgres is simply cheaper and more portable.
Auth OIDC + JWT bearer Works with whatever IdP the agency already has. We are not in the auth business.
Runtime Azure Container Apps Managed scale to zero, no Kubernetes ops overhead. Vendor coupling, abstracted behind IaC.
Observability OpenTelemetry Standardised, swappable backend (App Insights, Honeycomb, Grafana Cloud).
CI/CD GitHub Actions + Bicep Same repo, same review surface. Bicep over Terraform when we're Azure only.

How the architecture addresses non functionals

Security

  • OIDC into JWT bearer, short lived tokens, refresh on the gateway.
  • Secrets in Key Vault via managed identity. None in env files.
  • WAF in front, dependabot and Snyk on PRs, OWASP aware code review checklist.
  • Tenant scoping enforced at the repository layer.

Scalability

  • Stateless API, horizontal scale on CPU and RPS metrics.
  • Database read replica for reporting workloads.
  • Redis for hot reads (board view, user lookups).
  • Cursor pagination on list endpoints from day one.

Reliability

  • Health probes (liveness vs readiness).
  • Blue and green deploys with automated rollback on SLO burn.
  • Database PITR plus nightly logical backups off region.
  • Retries with jitter on outbound calls. Circuit breaker on third parties.

Observability

  • Structured logs (Serilog) with traceId on every request.
  • OpenTelemetry traces from browser through API to database.
  • RED metrics plus SLOs (latency, error rate, throughput) per endpoint.
  • Alert on burn rate, not on threshold spikes.

02

Delivery plan

Roadmap, team shape, the risks we'll watch, and how we'll know early.

Phased roadmap, prototype to launch

Now

Phase 0 · Foundation

  • Reference repo, ADRs, environments wired
  • Discovery workshops (users, integrations, compliance)
  • Auth provider and hosting confirmed
Wk 1 to 3

Phase 1 · MVP

  • Auth, multi user, tasks, assignment, due dates
  • CI/CD, staging, structured logs, traces
  • Manual UAT with three client users
Wk 4 to 6

Phase 2 · Beta

  • Comments, notifications, attachments
  • Real time updates (SignalR)
  • Performance pass, accessibility audit
Wk 7 to 9

Phase 3 · Launch

  • Security review, pen test remediation
  • SLOs and alerting in place, runbooks written
  • Phased rollout (10%, 50%, 100%)
Wk 10+

Phase 4 · Iterate

  • Feedback driven backlog
  • Tech debt budget (10 to 15% per sprint)
  • Quarterly architecture review

Team composition

Core, full time

  • Tech lead (me). Code, architecture, stakeholder management.
  • 2 full stack engineers (mid or senior). Vertical slice delivery.
  • Senior frontend with UX leaning. Pattern library, accessibility, performance.

Shared, part time

  • DevOps and platform (~50%). IaC, pipelines, on call setup.
  • Designer (~25%). Flows, design tokens, accessibility review.
  • QA (~50% from Phase 2). Exploratory and test automation tooling.

Client side

  • Product owner with real authority on scope and priority.
  • Two or three pilot users for weekly demos and UAT.

Why this shape

  • Small enough to fit one stand-up. Big enough for sustained pace.
  • Seniority spread enables mentorship without bottlenecks.
  • Specialists arrive when their work is the bottleneck, not before.

Top risks and how we'll mitigate them

RiskLikelihood / impactMitigation
Scope creep, the "just one more" High / Medium Strict MVP definition, change control via PO, weekly demos surface trade-offs early.
Auth provider undecided by client Medium / High OIDC abstraction means we can swap. Day one question. Default to Entra ID if undecided in week one.
Data migration from current ad hoc workflows Medium / High Discovery workshop in week one. If non trivial, build an importer as Phase 2, not as a launch blocker.
Hidden compliance requirement (GDPR, ICO) Low / High Privacy review with legal in week one. Design data model with PII flags and export/delete from day one.
Realtime becomes a launch must have Medium / Medium API designed around commands and queries. SignalR layers on without changing the data model.
Key person dependency Medium / High Pair programming, rotating PR reviewers, ADRs in repo, runbooks in repo.

How I'll surface delivery issues early

Cadence

  • Two week sprints, demo to PO and pilot users on Friday.
  • Daily 15 minute stand-up. Risks and blockers first, status second.
  • Weekly written status to client. RAG, top three risks, asks.

Engineering metrics

  • DORA: lead time, deploy frequency, MTTR, change failure rate.
  • PR cycle time and review latency, visible on a dashboard.
  • Flaky test budget enforced. No green on retry merges.

Product signals

  • Burn up of MVP scope vs target launch.
  • Weekly tag up with pilot users for qualitative feedback.
  • Post deploy error rate and latency dashboards reviewed each Monday.

Surfacing problems

  • RAG status is honest, not optimistic. Amber early beats red late.
  • If a sprint slips, the deliverable changes, not the date. We say so.
  • Blameless post mortems for incidents over 30 minutes. Written, shared, no individual named.

03

Technical leadership

Standards, contentious conversations, and what I'd ask on day one.

Setting the team up for success

Ways of working

  • Trunk based dev, short lived branches, daily merges.
  • Pair on anything risky or new. Mob occasionally for shared learning.
  • Tech debt time blocked weekly. Not optional.
  • Decisions documented as ADRs in repo, not in chat history.

Technical standards

  • Strict TypeScript, nullable reference types, warnings as errors.
  • OpenAPI is the API contract. Client types generated from it.
  • Every new endpoint ships with at least one integration test.
  • Logs are structured. No Console.WriteLine.

Code review

  • One approver for low risk. Two for prod paths, schema changes, or auth.
  • Author writes the PR description. Reviewer reads it before the diff.
  • Reviews are timely (target: 4 working hours). Slow review kills flow.
  • Comments separate blocking, nit, and question.

Definition of Done

  • Tests written and passing locally and in CI.
  • Deployed to staging. Observable in logs and traces.
  • Accessible (WCAG 2.1 AA spot check) and documented if user facing.
  • Feature flag in place if rollback isn't trivial.

"The PO wants this feature now, but it's risky"

  1. 1

    Acknowledge the goal.

    The feature exists because of a real outcome the PO needs. Start there, not at "no".

  2. 2

    Make the risk concrete.

    What breaks? Who's affected? What's the blast radius if it goes wrong? Numbers, not adjectives.

  3. 3

    Offer options, not roadblocks.

    A smaller scoped version. A time boxed spike. A flag protected rollout. Always two paths.

  4. 4

    Quantify "fast" vs "safe".

    Days, not feelings. The PO can choose with eyes open.

  5. 5

    If they still push.

    Write the decision and risks as an ADR. Get explicit sign-off. Ship behind a kill-switch with heightened monitoring.

  6. 6

    Never.

    Say "no" as the engineer alone. Or "yes" without the trade-off visible to whoever owns the budget.

What I'd ask on day one

About the users

  • Who actually uses this? Roles, team sizes, technical comfort?
  • What does a busy day look like for them today?
  • Which of the chaos (email, calls, meetings) do they actually want to replace, versus supplement?

About the data

  • Where does work live today? Spreadsheets, inboxes, project tools?
  • Does anything need importing on day one, or do we start fresh?
  • Are clients (the agency's clients) ever exposed to this app?

About hosting and identity

  • Existing cloud provider and identity (Entra, Google Workspace, Okta)?
  • UK or EU data residency required? PII categories present?
  • Any existing security or platform standards I should inherit?

About launch and success

  • What does v1 success look like? Measurable?
  • Who decides scope when there's pressure to add more?
  • What's the budget for ongoing operation post launch?

How I'd answer them in the first two weeks

Week 1

Workshops.

90 minute sessions with each user role. Observe a typical day on site or shadow call if remote.

Week 1

Written discovery.

Existing docs, policies, integration touch points. Security and IT alignment meeting.

Week 2

Wireframes.

Low fi mockups validated with pilot users before any further code beyond the prototype.

Week 2

ADRs published.

Auth provider, hosting region, data model approach. All written down and signed off.

Ongoing

Assumption log.

Anything we couldn't answer is logged with a target date for resolution.

Concerns I'd raise about the brief and prototype

"Production ready" is undefined.

We need explicit acceptance criteria: uptime, security, accessibility, observability targets.

No identity model in the prototype.

Single user code paths look simple. Real auth plus tenant scoping reshape almost every endpoint. We'd front load this.

No clear constraint on hosting or compliance.

Marketing data is personal data. GDPR is in scope by default.

The chaos to replace is bigger than tasks.

Email, calls, meetings imply notifications, mentions, summaries. Clarify what's in v1 versus Phase 2.

"Coordinate" is a multi app workflow.

Slack or Teams integration may be unavoidable. Worth confirming in week one.

In summary

"Production ready" isn't a checkbox.

It's a small team shipping every week, behind feature flags, with the trade-offs written down where everyone can see them.
